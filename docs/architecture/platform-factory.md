# Platform Factory

This repository is a reusable ROSA HCP factory built around Git-driven inputs, Terraform bootstrap, and OpenShift GitOps for day-2 configuration.

The target operating model is:

1. Users propose cluster inputs in Git.
2. A pull request is reviewed and approved.
3. CI validates and renders the effective configuration.
4. Merge to `main` runs Terraform to:
   - provision AWS prerequisites in scope
   - create the ROSA HCP cluster
   - register the cluster to ACM
   - bootstrap OpenShift GitOps
5. OpenShift GitOps reconciles all day-2 configuration using an App-of-Apps pattern.

The factory stays customer-neutral. Product names such as `cp4ba`, `aap`, `oadp`, and `mtv` are valid where they describe reusable workload or platform capabilities.

## End-to-End Flow

```text
Cluster class
  + cluster inputs
  -> render effective config
  -> Terraform modules
     -> ROSA HCP cluster
     -> ACM registration
     -> OpenShift GitOps bootstrap
        -> root application
           -> platform apps
           -> workload apps
```

## Terraform Scope

Terraform owns only cluster factory concerns:

- remote state
- AWS prerequisites in scope
- ROSA HCP cluster creation
- foundational machine pools
- ACM registration
- OpenShift GitOps bootstrap

## GitOps Scope

OpenShift GitOps owns all day-2 cluster state:

- identity providers
- RBAC
- logging and monitoring
- backup and restore
- secrets integration
- ingress and registry policies
- OpenShift Virtualization and MTV
- workload onboarding

## Repository Layout

- `catalog/`: reusable cluster and machine-pool classes
- `clusters/`: environment-specific cluster instances
- `modules/`: reusable Terraform modules
- `gitops/`: OpenShift GitOps bootstrap, applications, and overlays
- `scripts/`: CI-friendly helper scripts

## Engineer Workflow

```text
Engineer
  -> edit cluster.yaml and gitops.yaml
  -> open pull request

CI pipeline
  -> validate inputs
  -> render effective config
  -> run Terraform validation

After merge
  -> Terraform applies the stack
  -> Terraform seeds the OpenShift GitOps root app
  -> OpenShift GitOps reconciles platform and workload apps
```

## Audit Model

The source of truth is Git.

Human-authored inputs should be YAML. Generated artifacts should remain JSON.

Audit artifacts are generated by CI and should be stored as pipeline artifacts or in immutable object storage:

- effective config
- Terraform plan text and JSON
- provider/module versions
- outputs
- build metadata
